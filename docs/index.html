User
The auth middleware is rejecting valid refresh tokens after they've been rotated. Can you investigate and fix it?
Claude Opus 4
The user reports that valid refresh tokens are being rejected after rotation. This suggests a race condition in the token rotation logic — when a new token is issued, the old one should remain valid for a grace period to handle concurrent requests. Let me trace the flow: 1. Client sends refresh token 2. Server validates → issues new token pair 3. Old token is immediately invalidated 4. But if another request arrives with the old token before the client receives the new one... I need to look at the token rotation implementation.
I'll investigate the refresh token rotation logic. This sounds like a race condition where concurrent requests during rotation can fail.
Read — src/middleware/auth.rs:142-180
pub fn validate_refresh_token(token: &str) -> Result<Claims> { let claims = decode_jwt(token)?; // BUG: This immediately checks revocation without grace period if self.revoked_tokens.contains(&claims.jti) { return Err(AuthError::TokenRevoked); } // Rotate: revoke old, issue new self.revoked_tokens.insert(claims.jti.clone()); Ok(claims) }
Claude Opus 4
Found it. The validate_refresh_token function immediately revokes the old token with no grace period. When concurrent requests arrive during rotation, the second request sees the token as revoked.
